# Build Error Fixes Applied - April 8, 2026

## Summary
Fixed 2 major issues preventing successful builds and payment processing.

---

## Issue #1: DynamicServerError During Build (CRITICAL)

### Problem
Next.js build process was trying to statically pre-generate 85 pages, but 11 API routes used runtime-only features:
- `getServerSession()` → internally calls `headers()`
- `request.url` and `request.nextUrl.searchParams` → runtime-only APIs

These are NOT available during build-time static generation, causing the build to fail with `DynamicServerError: Page couldn't be rendered statically...`

### Root Cause
Next.js 13.5.11 attempts to find all data dependencies during build. Dynamic APIs raise errors instead of silently failing.

### Solution: Force Runtime-Only Routes
Added `export const dynamic = 'force-dynamic'` to the top of 11 API routes to instruct Next.js these must ALWAYS execute at runtime, never during build time.

### Fixed Routes (11 total)
1. ✅ `/api/admin/dashboard/route.ts` - Uses `getServerSession()`
2. ✅ `/api/admin/geocode-businesses/route.ts` - Uses `getServerSession()`
3. ✅ `/api/admin/property-listings/route.ts` - Uses `getServerSession() + searchParams`
4. ✅ `/api/business/my-businesses/route.ts` - Uses `getServerSession()`
5. ✅ `/api/business/property-listings/route.ts` - Uses `getServerSession() + searchParams`
6. ✅ `/api/businesses/route.ts` - Uses `request.url`
7. ✅ `/api/promotions/monthly-stats/route.ts` - Uses `searchParams`
8. ✅ `/api/property-listings/route.ts` - Uses `request.nextUrl.searchParams`
9. ✅ `/api/subscriptions/status/route.ts` - Uses `getServerSession() + searchParams`
10. ✅ `/api/user/dashboard/route.ts` - Uses `getServerSession()`
11. ✅ `/api/user/listings/route.ts` - Uses `getServerSession()`

### Expected Outcome
- ✅ Build completes successfully without DynamicServerError
- ✅ All 85 pages generate without deoptimization warnings  
- ✅ API routes execute normally at runtime

## Issue #2: PayGate 403 Blocking Error (CRITICAL)

### Problem
Payment checkout failing with:
```
PayGate HTTP 403: Request blocked
Generated by cloudfront (CloudFront)
```

Cause: CloudFront WAF blocking requests with incomplete/suspicious User-Agent header

### Solution: Proper Browser Headers
Updated `[src/lib/paygate.ts](src/lib/paygate.ts)` fetch headers:

**Changes:**
- ✅ Complete User-Agent: `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36`
- ✅ Standard browser headers: `Accept`, `Accept-Language`, `Accept-Encoding`
- ✅ Security headers: `Sec-Fetch-*` headers
- ✅ Cache control headers
- ✅ 30-second timeout for reliability

### Expected Outcome
- ✅ PayGate HTTP requests pass CloudFront inspection
- ✅ Payment checkout initiates successfully
- ✅ Proper error handling if PayGate API returns legitimate errors
