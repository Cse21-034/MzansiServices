# HTTP Request/Response Analysis - Payment Error

## Your Observed Error

```

Request Method: GET
Status Code: 200 OK
Remote Address: 216.198.79.65:443
Referrer Policy: strict-origin-when-cross-origin
Server: Vercel
```

This GET request with `error=payment_not_found` query parameter tells us **exactly where the error is happening**.

---

## Error Trace

### Where The Error Comes From

The `?error=payment_not_found` is generated by this code in `src/app/api/subscriptions/return/route.ts`:

```typescript
// Line 27-35
if (!payment) {
  console.error('[Return Handler] Payment not found for payRequestId:', payRequestId);
  return NextResponse.redirect(
    new URL('/?error=payment_not_found', request.nextUrl.origin),  // ← This line
    { status: 303 }
  );
}
```

### Why It Triggers

The return handler receives a POST request from PayGate with `PAY_REQUEST_ID` and tries:

```typescript
const payment = await prisma.payment.findUnique({
  where: { payRequestId },  // ← Looks for payment with this ID
});
```

But the database has:
```
Payment { 
  payRequestId: null  // ← But it's null! So findUnique returns null
}
```

So the condition `if (!payment)` becomes true, and it returns the error redirect.

## Why The Error Occurs - Side By Side Comparison

### What Happens (Broken) ❌
```
Request 2 (Checkout)       → Response with params ✅
Request 3 (Initiate)       → Response with PAY_REQUEST_ID ❌ Not extracted
[Skipped] Save Request     → Never called
Request 4 (Process)        → MAY Not called
Request 5 (Payment)        → ✅ Payment processes successfully!
Request 6 (Callback)       → ✅ Updates payment by transactionRef
                           → Database: payRequestId = null
Request 7 (Return)         → ❌ Tries to find by payRequestId
                           → payRequestId is null
                           → Query returns no results
                           → Returns error redirect
Request 8 (Home Error)     → ✅ Shows: ?error=payment_not_found
```

### What Should Happen (Fixed) ✅
```
Request 2 (Checkout)       → Response with params ✅
Request 3 (Initiate)       → Response with PAY_REQUEST_ID ✅ EXTRACTED
Request 3B (Save)          → Save PAY_REQUEST_ID to DB ✅
                           → Database: payRequestId = "abc123..."
Request 4 (Process)        → ✅ Called with extracted ID
Request 5 (Payment)        → ✅ Payment processes
Request 6 (Callback)       → ✅ Updates payment
                           → Database: payRequestId stays set
Request 7 (Return)         → ✅ Finds by payRequestId
                           → Query returns the payment
                           → Updates status
Request 8 (Success)        → ✅ Redirects to: /business/{id}/subscription/success
```

## The Fix In One Sentence

**The frontend needs to capture the `PAY_REQUEST_ID` from PayGate's initiate.trans response and save it to the database before submitting to process.trans.**

That's it. Everything else works perfectly!
